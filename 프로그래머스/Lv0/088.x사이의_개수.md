# x 사이의 개수

## 문제 설명
문자열 `myString`이 주어집니다. `myString`을 문자 "x"를 기준으로 나눴을 때 나눠진 문자열 각각의 길이를 순서대로 저장한 배열을 return 하는 solution 함수를 완성해 주세요.

## 제한사항
- 1 ≤ `myString`의 길이 ≤ 100,000
    - `myString`은 알파벳 소문자로 이루어진 문자열입니다.
    
## 입출력 예
| myString | result |
| :-- | :-- |
| "oxooxoxxox" | [1, 2, 1, 0, 1, 0] |
| "xabcxdefxghi" | [0, 3, 3, 3] |

## 입출력 예 설명
입출력 예 #1
- "x"를 기준으로 문자열을 나누면 ["o", "oo", "o", "", "o", ""]가 됩니다. 각각의 길이로 배열을 만들면 [1, 2, 1, 0, 1, 0]입니다. 따라서 [1, 2, 1, 0, 1, 0]을 return 합니다.

입출력 예 #2
- "x"를 기준으로 문자열을 나누면 ["", "abc", "def", "ghi"]가 됩니다. 각각의 길이로 배열을 만들면 [0, 3, 3, 3]입니다. 따라서 [0, 3, 3, 3]을 return 합니다.

## 이전 학습 내용 적용
- 직전에 공부했던 `split`을 사용해서 문제를 해결하려고 했지만 공백의 문자열도 카운트를 해야한다는것을 깨달았고 그에 맞춰 `split`과 같이 공부했던 `components`를 사용하여 문제를 해결하였다.

## 배운점
- `split(separator:, omittingEmptySubsequences:)`
    - `components`는 데이터가 많을 경우 임시 문자 배열을 생성하여 메모리의 효율이 떨어진다.
    - `split`의 파라미터 `omittingEmptySubsequences`의 값을 false로 주어 `Substring`의 가벼운 메모리를 활용하며 빈문자열도 나오게 할수 있다는것을 알게되었다.
    - 또한 최종 반환값이 문자열이 아닌 Int로 인해 `map { $0.count }`를 거치는 순간 원본 문자열과의 메모리가 자연스럽게 끊어진다는 것을 확인했다.
    
- 컴파일러 Ambiguous 오류
    - 위에 처럼 `omittingEmptySubsequences`를 사용하자 `Ambiguous use of 'split'` 오류가 발생하였다.
    - `String`이 1차원적 흐름인 `Sequence`와 Index 접근이 가능한 `Collection` 프로토콜을 동시에 준수하기 때문이다.
    - 이에 컴파일러에서 `Sequence.split` -> `ArraySlice`를 사용해야 할지 `Collection.split` -> `Substring`을 사용해야 할지 몰라 오류가 발생하였다.
    
- 역방향 타입 추론 활용
    - 이 커파일러 오류를 해결하기 위해 `map { $0.count }`의 클로저 안에 `map { (str: Substring) in str.count }`와 같이 타입을 명시해 주었다.
    - 이를 통해 컴파일러에게 `Substring`을 반환하는 `Collection.split`을 사용하도록 역방향 타입 추론을 유도하는 방법을 알게 되었다.

## 나의 풀이

### 첫 번째 풀이
```swift
func solution(_ myString:String) -> [Int] {
    myString.components(separatedBy: "x").map { $0.count }
}
```

### 두 번째 풀이
```swift
func solution(_ myString:String) -> [Int] {
    myString.split(separator: "x", omittingEmptySubSequences: false).map { (str: Substring) in str.count}
}
```
